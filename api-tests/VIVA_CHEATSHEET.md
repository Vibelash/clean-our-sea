## 1. Thunder Client — the demo flow (≈3 minutes)

Run these in order. They tell one story: "user plays the game, takes a
quiz, the unified leaderboard reflects both."

### 1.1 GET — full news feed (proves the seeder ran)

```
GET  http://localhost:8080/news
```

Talk: "Backend boots, NewsSeeder seeds 15 SDG-14 articles into H2
once, idempotent on restart. This GET is what powers `news.html`."

### 1.2 GET — keyword search (non-trivial backend logic)

```
GET  http://localhost:8080/news/search?q=ocean
```

Talk: "Spring Data JPA query — case-insensitive LIKE across title,
summary AND body. Empty queries return the full feed; single-letter
queries return 400 to avoid expensive scans."

### 1.3 GET — category aggregation (GROUP BY)

```
GET  http://localhost:8080/news/categories
```

Talk: "GROUP BY in the repo, mapped to a CategoryCountDTO in the
service. Returns `[{category, count}, …]`."

### 1.4 POST — create a news article (CREATE)

```
POST http://localhost:8080/news
Content-Type: application/json

{
  "title":   "Demo article — created during the viva",
  "summary": "Created live to demonstrate POST.",
  "body":   "Full body text — this row was generated by the backend.",
  "category":"action",
  "icon":   "🌊",
  "source": "Karan — viva demo",
  "sourceUrl": null
}
```

**Save the returned `id`** — you'll use it in the next four steps.

### 1.5 PUT — replace mutable fields (UPDATE)

```
PUT  http://localhost:8080/news/{id}
Content-Type: application/json

{
  "title":   "Updated headline",
  "summary": "Updated summary text shown on cards.",
  "body":   "Updated full body — replaced via PUT.",
  "category":"science",
  "icon":   "🔬",
  "source": "Updated source",
  "sourceUrl":"https://example.com"
}
```

Talk: "PUT is the missing fourth verb on the news resource. It
replaces the editorial fields but never touches `likes` — likes have
their own atomic endpoint."

### 1.6 PATCH — atomic +1 like (the non-trivial bit examiners look for)

```
PATCH http://localhost:8080/news/{id}/like
```

Click it twice, point at `likes` going 1 → 2.

Talk: "PATCH because liking is a small atomic mutation of one field.
The repo runs `UPDATE news_post SET likes = likes + 1 WHERE id = ?`
inside a `@Transactional` service — two simultaneous likes can't
clobber each other."

### 1.7 DELETE — remove the demo article (DELETE)

```
DELETE http://localhost:8080/news/{id}
```

Then GET it again → 404.

### 1.8 POST — submit a snake score for Alex (cross-module link)

```
POST http://localhost:8080/scores
Content-Type: application/json

{ "player": "Alex", "userId": 1, "score": 320 }
```

Talk: "Frontend sends the userId from `localStorage.cleanOurSea.user`.
ScoreService.saveScore looks up that User and bumps `totalScore` and
`weeklyPoints` — that's how Tala's leaderboard reflects gameplay
automatically."

### 1.9 GET — unified cross-module leaderboard (THE A-grade slide)

```
GET  http://localhost:8080/api/leaderboard
```

Point out Alex's row — you just bumped him. Combined score = totalScore
(Tala's table) + bestQuizScore (Daniel's table). bestSnakeScore is the
top single Score row (Karan's table).

Talk: "This is the single endpoint that proves all five teammate
backends are *one* Spring Boot application and *one* H2 database. It
reads three repositories — `UserRepository` (Tala), `ScoreRepo`
(Karan), `UserQuizRepository` (Daniel) — joins them in one service,
returns one JSON. Read-only, no schema added: the integration was
already there at the data layer; this just surfaces it."

### 1.10 GET — SDG-14 environmental impact (thematic tie-back)

```
GET  http://localhost:8080/scores/impact
```

Talk: "Closes the loop with the problem statement: total in-game points
÷ 10 = pieces of plastic notionally cleaned. That's the SDG 14 story."

## 2. H2 console — copy/paste queries

Open `http://localhost:8080/h2-console` → JDBC `jdbc:h2:file:~/cleanoursea`
→ user `sa` → Connect.

### 2.1 Show every shared table — "one DB, eleven tables"

```sql
SHOW TABLES;
```

Expected: COMMUNITIES, COMMUNITY_TAGS, NEWS_POST, POLLUTION_REPORT,
POSTS, QUESTIONS, QUIZZES, REPORTS, SCORE, USERS, USER_QUIZZES.

### 2.2 The cross-module integration query (one row per real user)

```sql
SELECT u.id,
       u.username,
       u.total_score                                  AS leaderboard_total,
       (SELECT MAX(s.score) FROM score s
         WHERE s.user_id = u.id)                      AS best_snake,
       (SELECT MAX(uq.score) FROM user_quizzes uq
         WHERE uq.user_id = u.id)                     AS best_quiz,
       (SELECT COUNT(*)    FROM score s
         WHERE s.user_id = u.id)                      AS games_played
FROM users u
ORDER BY u.total_score DESC;
```

This is the SQL behind `GET /api/leaderboard`. Same data, two ways:
JSON for the frontend, SQL for the marker.

### 2.3 Recent snake games (proves Frontend → Backend → DB)

```sql
SELECT s.id, s.player, s.score, s.user_id, u.username
FROM score s LEFT JOIN users u ON u.id = s.user_id
ORDER BY s.id DESC
LIMIT 10;
```

After Alex's POST in step 1.8 you'll see his row at the top.

### 2.4 News-side state (proves PATCH likes really mutated the DB)

```sql
SELECT id, title, category, likes, posted_at
FROM news_post
ORDER BY likes DESC, posted_at DESC
LIMIT 10;
```

The article you liked twice in 1.6 is at the top with `LIKES=2`. After
you DELETE it in 1.7, run this again — it's gone.

### 2.5 News categories — same data, different lens

```sql
SELECT category, COUNT(*) AS posts
FROM news_post
GROUP BY category
ORDER BY category;
```

Matches the JSON from `GET /news/categories` exactly. Tells the
examiner the aggregation isn't faked in the service.

### 2.6 The whole "cross-module link" in one screen

```sql
SELECT u.username,
       u.email,
       u.total_score       AS leaderboard_total,
       (SELECT COUNT(*) FROM score        WHERE user_id = u.id) AS snake_games,
       (SELECT COUNT(*) FROM user_quizzes WHERE user_id = u.id) AS quizzes_taken
FROM users u
WHERE u.email IS NOT NULL
ORDER BY u.total_score DESC;
```

`email IS NOT NULL` filters to users who actually registered through
`/auth/register`, separating real authenticated users from seeder rows.

## 3. Quick failure-case demos (one-liners for Q&A)

If the examiner asks "what about bad input?":

```
GET    http://localhost:8080/news/search?q=a       # too short → 400
POST   http://localhost:8080/scores                # negative   → 400
       { "player": "x", "score": -5 }
GET    http://localhost:8080/scores/player/Nobody  # missing    → 404
PATCH  http://localhost:8080/news/99999/like       # missing id → 404
```

All four are validated by the service layer (`ResponseStatusException`),
not the controller — so 400/404 reach the frontend with a real message.

## 5. The 30-second answers (anticipated questions)

- **"Why PATCH for the like, not POST?"** PATCH expresses a partial
  mutation of one field. POST would imply creating a new resource.
- **"Why is `likes` `BIGINT NOT NULL DEFAULT 0`?"** So existing rows
  survive the schema migration when Hibernate adds the column to a
  populated H2 file. Without the default, ALTER TABLE failed with
  "NULL not allowed".
- **"What stops two users liking simultaneously and losing one?"**
  `@Modifying @Query("UPDATE … SET likes = likes + 1 …")` runs as a
  single SQL UPDATE inside a `@Transactional` boundary. It's atomic.
- **"Where's the integration evidence?"** `GET /api/leaderboard` —
  one endpoint, three teammates' tables, one JSON. The H2 query in
  §2.2 is the same join in SQL.
- **"How is this tested?"** 21 JUnit cases (in-memory H2) plus a
  40-assertion curl smoke test against the real backend. Both green
  on `mvn test` and `bash curl-tests.sh`.
